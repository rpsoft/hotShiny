● I've completed the examination. Here's my assessment and proposal.

  Where compatibility stands today

  The API surface coverage is actually decent: most UI tags, inputs, outputs,
  layouts, navigation, modals, and update functions are exported, and they reuse
  Shiny's S3 class names (shiny.tag, shiny.tag.list), which is the right call.
  The faithful-app example works. But "swap library(shiny) for
  library(hotShiny)" fails today for almost any real app, for four structural
  reasons rather than a long tail of missing functions:

  1. session is NULL (R/app.R:498). Every server function is function(input, 
  output, session), and a huge fraction of real apps touch session —
  session$sendCustomMessage, session$ns, session$userData,
  session$onSessionEnded, session$clientData, observe(session$reload()).
  Ironically your own package already needs it: R/ui-dynamic.R, R/ui-modals.R
  and R/ui-updates.R all call session$sendCustomMessage(...) on an object that
  is always NULL, so insertUI, showModal, showNotification etc. can currently
  never work when called the Shiny way.

  2. Input values have wrong types and wrong null semantics
  (R/core-values.R:120-188). InputProxy$get returns "" for unset inputs and
  coerces everything through a character→numeric heuristic. In Shiny, input$x is
  NULL until the client reports it, checkboxInput yields TRUE/FALSE (logical),
  checkboxGroupInput yields a character vector, sliderInput with value = c(1,10)
  yields length-2 numeric, fileInput yields a data frame, dateInput yields a
  Date. Any app doing if (is.null(input$x)), req(input$x), or if (input$chk)
  breaks. This single file probably breaks more legacy apps than anything else,
  and fixing it doesn't conflict with the graph model at all.

  3. Static AST dependency extraction can't see everything. Dependencies are
  extracted from the expression tree at registration time (core-reactive.R,
  core-observe.R). That misses: input[[paste0("sel_", i)]], reactives created in
  lapply/loops, dependencies hidden inside helper functions defined elsewhere,
  and module indirection. Shiny discovers dependencies at runtime, so these
  patterns are common in the wild. This is the genuinely "impossible without
  translation" category — more below. (Silver lining: isolate() is trivially
  implementable in your model — just skip dependency extraction inside isolate()
  subtrees — yet it's currently not exported at all.)

  4. The client JS doesn't speak Shiny's protocol. inst/www/inputs.js uses a
  custom hotShiny.wsClient contract. The entire htmlwidgets ecosystem (DT,
  plotly, leaflet, …) and shinyjs depend on the browser-side Shiny object:
  Shiny.setInputValue(), Shiny.addCustomMessageHandler(),
  Shiny.outputBindings.register(), Shiny.inputBindings.register(). Without a
  compat shim, no ecosystem package's JavaScript can ever bind.

  There's also a fifth, self-inflicted issue: the sys.source/load_env machinery
  throughout app.R (lines 7–117, 380–490, 945–1103) re-sources package files
  from getwd(). That only works when the app happens to run from the package
  source directory — an installed hotShiny running a user's app from their own
  directory falls into fragile fallback paths. This isn't a Shiny-compat issue
  per se, but it will dominate "it doesn't work on my machine" reports and
  should be replaced with normal namespace resolution before any compat work
  lands on top of it.

  Proposed work, in priority order

  Tier 1 — Make honest drop-in swap possible for simple apps

  1. shinyApp(ui, server) alias for app(), plus shinyServer/shinyUI legacy
  shims, stopApp(), and make runApp() accept a directory (Shiny's signature)
  containing app.R or ui.R/server.R + global.R. Today runApp() requires a file
  that defines bare ui/server objects — a real legacy app ends with shinyApp(ui,
  server), which would currently error.
  2. Fix input semantics — NULL until set, preserve JSON types from the
  websocket (logical, numeric, vectors, dates), per the gap described above.
  3. A real session object. An R6 ShinySessionProxy passed as the third server
  argument, implementing the documented surface: sendCustomMessage (wired to
  your existing ws handlers — this instantly makes your own
  modals/notifications/dynamic-UI work), ns (identity for now), userData,
  onSessionEnded/onFlush/onFlushed, clientData (a reactiveValues fed by the
  client), input/output, reload, sendInputMessage. getDefaultReactiveDomain()
  should return it.
  4. Missing core reactive verbs, roughly in frequency-of-use order: isolate
  (easy in your model), reactiveVal, eventReactive, req/validate/need + the
  shiny.silent.error condition system, invalidateLater/reactiveTimer, bindEvent,
  debounce/throttle, reactiveValuesToList, is.reactive, freezeReactiveValue,
  reactivePoll/reactiveFileReader, exprToFunction/installExprFunction (needed by
  htmlwidgets later).
  5. Graceful failure for everything else. Generate stubs for every remaining
  shiny export that hotShiny lacks, each throwing a structured, catchable error:
  "bookmarking is not yet supported in hotShiny — see the migration guide / run
  shiny2hotshiny::check_app()". This converts mysterious could not find 
  function "enableBookmarking" into actionable diagnostics, and gives you a
  machine-readable inventory of the gap (useful for the translator's lint mode
  too). A small script that diffs getNamespaceExports("shiny") against your
  NAMESPACE can generate and maintain this.

  Tier 2 — Modules and ecosystem foundations

  6. Modules: NS(), moduleServer(), callModule(). Almost all non-trivial modern
  Shiny code uses modules. The mapping is natural in your model — a namespace is
  just a prefix on node IDs (input.mymod-x). NS is pure string manipulation;
  moduleServer needs to run the module function with namespaced input/output
  proxies.
  7. Depend on htmltools instead of reimplementing it. ui-tags.R (1,375 lines)
  re-creates htmltools. Ecosystem packages return real htmltools objects
  carrying htmlDependency attachments; your renderer must run
  htmltools::findDependencies() + resolveDependencies() and emit <script>/<link>
  into the page head, and addResourcePath() must exist to serve them. Swapping
  your tag layer for htmltools (keeping your tag_to_html only as the
  diff-renderer entry point, via htmltools::renderTags) removes a whole class of
  subtle incompatibilities at once.
  8. A Shiny JS compat shim (shiny-compat.js): implement Shiny.setInputValue,
  Shiny.addCustomMessageHandler, Shiny.inputBindings/outputBindings.register,
  Shiny.bindAll/unbindAll, and the shiny:connected/shiny:value events on top of
  your websocket client. With #7 + #8 + registerInputHandler() on the R side,
  shinyjs and simple htmlwidgets start working without those packages changing
  anything.

  With Tier 2, a realistic target is: shinyjs, DT, plotly, leaflet (htmlwidgets 
  generally), shinyWidgets mostly working; bslib/shinydashboard partially
  (they're heavy on htmltools theming and expect Shiny's exact Bootstrap markup
  — note you ship Bootstrap 5 while classic Shiny markup is Bootstrap 3, which
  will bite shinydashboard hard; document it as unsupported initially).

  Tier 3 — The translator, for what's genuinely impossible

  Static graph extraction means some Shiny patterns can never just run. That's
  where the shiny2hotshiny translator comes in. I'd build it as a separate small
  package (or inst/translator/) with two modes:

  - check_app(path) (lint mode) — parse the app with parse(keep.source = TRUE),
  walk the AST, and emit a report classifying every finding as ✅ compatible /
  🔧 auto-translatable / ❌ needs manual rewrite, with file:line locations. This
  is the cheap, high-value first deliverable: users learn in seconds whether
  their app can move.
  - translate_app(path, out) (rewrite mode) — apply mechanical rewrites and
  insert # TODO(hotShiny): comments at the impossible sites.

  Auto-translatable patterns: library(shiny) → library(hotShiny); shinyApp(ui, 
  server) → app(ui, server) (until the alias from Tier 1 makes this
  unnecessary); callModule(f, id) → moduleServer form; deprecated
  renderDataTable → DT:: form; tags$script inline JS → flagged.

  Needs-manual-rewrite patterns (detect and explain, don't attempt):
  - Computed input names: input[[paste0(...)]], input[[var]] — invisible to
  static dep extraction. Suggested rewrite: enumerate the ids explicitly or use
  a documented hotShiny escape hatch (you could offer depends_on(c("a","b"), 
  expr) as an explicit-dependency annotation — worth adding to the package as
  the official workaround).
  - Reactives/observers created in loops or lapply over runtime data.
  - Dependencies inside externally defined helper functions (the dep extractor
  can't follow source()d code unless you teach it to inline known function
  bodies — a possible later enhancement).
  - Side-effectful patterns your strict mode already detects (<<- into shared
  state, non-determinism) — reuse core-strict-mode.R's detectors in the linter
  so the two stay consistent.
  - session$registerDataObj, bookmarking, renderCachedPlot cache keys,
  promises/ExtendedTask async.

  Cross-cutting: a compatibility test harness

  Wire Shiny's canonical example apps (system.file("examples", package = 
  "shiny") — 01_hello through 11_timer) into your existing tests/puppeteer
  setup, asserting they run unmodified under hotShiny. That suite is the
  definition of "drop-in compatible," makes regressions visible, and gives the
  README an honest scoreboard ("9/11 Shiny examples pass").

  Suggested sequencing

  Tier 1 items 1–3 (app entry points, input semantics, session object) are the
  highest leverage — they're prerequisites for everything else, they fix bugs in
  hotShiny's own exported functions, and none conflict with the graph model.
  The namespace-loading cleanup in app.R should happen first or alongside, since
  the session/input work touches the same plumbing. The check_app() linter is
  independent and could be started anytime — it's also the cheapest way to learn
  from real-world apps which gaps actually matter.

  Happy to turn this into a tracked plan (e.g., a COMPATIBILITY.md roadmap in
  the repo, or start on Tier 1) — just say which piece you'd like first.

● Good idea — the reactiveVal/reactiveValues write problem is the central one.
  Let me think about it properly.

  The root cause: the executor serializes each reactive/observer to an AST, then
  reconstructs and evaluates it in a synthetic environment parented to
  globalenv(), with reactive deps injected as no-arg getters. That throws away
  the live closure — so the rv function (reactiveVal) and the live
  reactiveValues R6 object aren't reachable, and writes can't round-trip.

  parent.frame() at the observe()/reactive()/render*() call site — i.e. the
  server body). The practical fix is to parent the eval environment to node$env
  instead of globalenv(), then overlay the input proxy and reactive getters on
  top. Live rv, reactiveValues objects, and user helper functions then resolve
  naturally, while the overlays still solve the "named reactive isn't callable"
  problem.

  The reactiveVal/reactiveValues write problem — solved

  Root cause: the executor serialized each reactive/observer to an AST and
  evaluated the reconstruction in a synthetic environment parented to
  globalenv(), with deps injected as no-arg getters. That discarded the live
  closure, so the rv function and the live reactiveValues object were
  unreachable — writes couldn't round-trip.

  The practical fix (two parts):

  1. Parent the eval environment to the node's captured closure (node$env)
  instead of globalenv(), in all three execute paths. The closure is the server
  body, where rv, reactiveValues objects, and user helpers actually live — so
  rv(rv() + 1) and vals$n <- vals$n + 5 resolve to the real objects and writes
  work. The input proxy and reactive getters are still overlaid on top, so
  named-reactive greeting() callability is preserved. This also fixed a latent
  gap: user-defined helper functions/variables in the server body are now 
  reachable by reactive expressions (they weren't before).
  2. Deferred, coalesced invalidation. Because reactive-value reads aren't
  static dependency edges, a write triggers a conservative refresh of all
  reactive/render nodes. Running that inline re-fired the writing observer
  (counter incremented by 2 per click). I made the refresh defer to the next 
  event-loop tick (after dirty flags clear) and coalesce — so observers don't
  re-fire and writes are exact. Observers are deliberately excluded from the
  refresh to avoid spurious side effects and infinite loops.
